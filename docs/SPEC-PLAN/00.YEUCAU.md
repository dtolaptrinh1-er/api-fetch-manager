# API Fetch Manager — Tổng hợp yêu cầu dự án

> Tài liệu tổng hợp toàn bộ yêu cầu dự án dưới góc nhìn kỹ thuật + quản lý product. Nguồn: page `api-fetch-manager` và `api-fetch-manager-json`.

---

## 1\. Tổng quan sản phẩm

Hệ thống quản lý tập trung **API key của nhiều dịch vụ** (GitHub, Supabase, Cloudflare, Tailscale, dpdns...), gắn theo từng `emailOwner`. Cho phép **thực thi API call bằng key đã lưu** (tạo repo, chạy workflow...), có lịch sử + log chi tiết để agent tự debug, và cơ chế mở rộng API kiểu **fetch builder** (input curl → chọn field lưu → output).

---

## 2\. Yêu cầu chức năng cốt lõi

1. **Quản lý key**: 1 key có thể có nhiều giá trị khác nhau, gom theo `emailOwner`. Hỗ trợ import/export dữ liệu.
2. **Luồng chính**: chọn `emailOwner` → chọn loại API → gọi API → lưu kết quả.
3. **Fetch Builder**: thêm API mới bằng cách paste lệnh `curl`, khai báo field cần lưu → sinh endpoint tái sử dụng (từ đầu vào → output).
4. **Lịch sử gọi API**: lưu toàn bộ lịch sử request/response.
5. **Log chi tiết**: để agent trace lỗi (theo nghiệp vụ, theo API server).
6. **Inspect Element Mode** (chế độ riêng): bật inspect → chọn nhiều element → mở form nhập tiêu đề / mô tả / kết quả mong muốn (kèm info element) → lưu RTDB (có trạng thái, ngày). Có form quản lý CRUD bug/issue. Xuất/copy Markdown cho agent thực hiện theo.
7. **Quản lý issue code**: bug, feature...

---

## 3\. Lưu trữ & kiến trúc dữ liệu

- **Google Realtime Database (RTDB)**, tách nhiều database độc lập:
  - 1 RTDB cho quản lý key
  - 1 RTDB riêng cho lịch sử gọi API
  - (tách thêm theo nghiệp vụ khi cần)
- Cấu trúc dữ liệu **có index** để truy vấn nhanh nhất.

---

## 4\. Môi trường triển khai

- Self-host
- GitHub Actions runner (Linux)
- Azure runner (Linux)
- Triển khai qua **Docker**
- **Toàn bộ biến môi trường bắt buộc prefix**: `API_FETCH_MANAGER_xxxxxxxxxx`

## 5\. Yêu cầu UI/UX (tóm tắt — chi tiết ở SPEC_UI)

- Responsive desktop + mobile, thống nhất toàn hệ thống
- Theme sáng / tối
- Font mảnh, khoảng cách gần
- Mọi thông báo qua **modal form**, không dùng `alert` của browser
- Mọi form tự detect chiều cao/rộng để tạo scrollbar (kể cả form thông báo, modal input)
- Modal phải có nút thoát; **click ra ngoài không đóng**
- Button phải có icon + tooltip mô tả chức năng & kết quả
- Chức năng quan trọng phải có confirm xác nhận
- Phải có file SPEC thiết kế riêng

## 7\. Bổ sung yêu cầu — Fetch nâng cao & tái sử dụng dữ liệu (v1.1)

### 7.1 Fetch Template nâng cao\\n- **Trích response đa cấp**: parse theo placeholder nhiều cấp, mảng, key (dùng JSONPath: `$.data.items[0].id`, wildcard `[*]`).\\n- **Sequential fetch (flow nhiều step)**: 1 lần gọi chạy tuần tự nhiều fetch, ví dụ lấy access token ở step 1 → truyền sang API step 2. Có shared context giữa các step, dừng khi step lỗi.\\n- **Input runtime**: truyền input trong lúc gọi, tùy cấu hình fetch builder (hỏi qua modal khi execute).\\n\\n### 7.2 UI quản lý dữ liệu trích xuất\\n- Màn quản lý các API có trích dữ liệu trong quá trình fetch. Ví dụ: tạo repo → lưu tên repo + link git; tạo nameserver Cloudflare → lưu id... Xem được template nào trích field gì và giá trị đã trích theo emailOwner + thời điểm.\\n\\n### 7.3 Kho placeholder tái sử dụng\\n- Lưu giá trị các placeholder để dùng lại nhiều lần (`{{var.xxx}}`). Giá trị trích xuất có thể pin vào kho này.\\n\\n### 7.4 JavaScript nhẹ trên placeholder\\n- Hỗ trợ transform: `upper`, `lower`, `trim`, `replace`, `base64`, `date`... dạng pipe `{{repoName | upper}}`.\\n- Hỗ trợ cả JS expression tùy ý chạy trong sandbox cô lập (cấm network/fs, timeout).
