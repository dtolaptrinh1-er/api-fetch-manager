# API Fetch Manager — PLAN triển khai
> Kế hoạch triển khai chi tiết, chia bước. Mỗi bước map tới yêu cầu cụ thể trong SPEC (tên SPEC + số mục), có **Definition of Done (DoD)**, **report hoàn thành**, **kết quả code cần có (file/thành phần)**. Mục tiêu: agent tự tick checklist, con người vào kiểm tra được.
**Version:** 1.0 · **Ngày:** 2026-07-07

**Ký hiệu SPEC:**
*   **\[REQ\]** = page _Tổng hợp yêu cầu dự án_
*   **\[SYS\]** = page _SPEC tổng thể (System Spec)_
*   **\[UI\]** = page _SPEC\_UI (Design Skill)_

**Quy ước report mỗi bước:** khi xong, agent ghi vào page **PROMPT (Nhật ký triển khai)** theo mẫu: `Bước X — trạng thái — file tạo/sửa — cách test — ghi chú/lưu ý`.

## Phase 0 — Khởi tạo & nền tảng
### Bước 0.1 — Scaffold monorepo + Docker
*   **Map SPEC:** \[SYS\] 2, 2.1, 8; \[REQ\] 4.
*   **Thành phần:** repo structure (`/frontend`, `/backend`, `/docs`, `/docker`), Dockerfile multi-stage, docker-compose, config TS cho cả FE/BE.
*   **Kết quả code cần có:**
    *   `Dockerfile` (multi-stage: build FE → copy vào BE → 1 image)
    *   `docker-compose.yml` chạy bằng env prefix `API_FETCH_MANAGER_`
    *   `backend/tsconfig.json`, `frontend/tsconfig.json`, `package.json` mỗi bên
    *   `.env.example` (khung, sẽ điền ở 0.2)
*   **DoD:** `docker compose up` chạy được container rỗng, health-check `GET /api/health` trả `{ ok: true }`.
*   **Report:** ảnh/log container chạy + kết quả curl health.
### Bước 0.2 — .env & config loader
*   **Map SPEC:** \[SYS\] 7.2; \[REQ\] 4 (prefix `API_FETCH_MANAGER_`).
*   **Kết quả code cần có:**
    *   `backend/src/config/env.ts`: load + validate mọi biến, fail-fast nếu thiếu.
    *   `.env.example` đầy đủ **có comment từng biến** (xem docs vận hành, Bước docs).
*   **DoD:** thiếu biến bắt buộc → app dừng với message rõ; mọi biến đều có prefix.
*   **Report:** danh sách biến + trạng thái validate.
### Bước 0.3 — Kết nối RTDB (multi-DB) + crypto
*   **Map SPEC:** \[SYS\] 2, 3, 7.1; \[REQ\] 3.
*   **Thành phần:** Firebase Admin SDK khởi tạo 5 DB (keys, history, logs, issues, variables); module crypto AES-256-GCM.
*   **Kết quả code cần có:**
    *   `backend/src/db/rtdb.ts` (khởi tạo & export 5 handle DB)
    *   `backend/src/lib/crypto.ts` (`encrypt/decrypt` AES-256-GCM, key từ env)
    *   Realtime Database Rules mẫu có `.indexOn` đúng schema mỗi DB
*   **DoD:** ghi/đọc thử mỗi DB thành công; encrypt→decrypt round-trip pass unit test.
*   **Report:** kết quả test round-trip + xác nhận 5 DB kết nối.

## Phase 1 — Backend core
### Bước 1.1 — Owner & Credential API (mã hoá at-rest)
*   **Map SPEC:** \[SYS\] 3.1, 4.1, 7.1; \[REQ\] 2.1.
*   **Kết quả code cần có:**
    *   `backend/src/modules/owners/*` (routes, service, repo)
    *   `backend/src/modules/credentials/*` (CRUD, mã hoá trước khi lưu, trả về **masked**)
    *   `backend/src/modules/data/import-export.ts`
*   **DoD:**
    *   Tạo owner, thêm nhiều giá trị cho cùng 1 key.
    *   Giá trị lưu trong RTDB là ciphertext (kiểm tra trực tiếp DB).
    *   API list trả masked (`ghp_****7ANR1M`), không bao giờ plaintext.
    *   Import/export round-trip giữ nguyên dữ liệu.
*   **Report:** payload mẫu + ảnh DB cho thấy ciphertext + response masked.
### Bước 1.2 — Placeholder Engine (transform + JS sandbox)
*   **Map SPEC:** \[SYS\] 10.6; \[REQ\] 7.4.
*   **Kết quả code cần có:**
    *   `backend/src/engine/placeholder.ts` (resolve order: credential → var → ctx → input → transform)
    *   `backend/src/engine/transforms.ts` (whitelist: upper, lower, trim, replace, slice, base64, base64decode, jsonStringify, urlEncode, date, default)
    *   `backend/src/engine/sandbox.ts` (isolated-vm, cấm network/fs, timeout 200ms)
*   **DoD:** pipe `{{x | upper}}` đúng; advanced JS chạy sandbox, không truy cập được network/fs, timeout hoạt động; có unit test cho từng transform.
*   **Report:** bảng kết quả từng transform + test sandbox chặn network/fs/timeout.
### Bước 1.3 — rtdb-variables (kho placeholder)
*   **Map SPEC:** \[SYS\] 10.4; \[REQ\] 7.3.
*   **Kết quả code cần có:**
    *   `backend/src/modules/variables/*` (CRUD, scope global + owner, `.indexOn`)
*   **DoD:** CRUD biến; tham chiếu `{{var.key}}` resolve đúng (ưu tiên owner, fallback global).
*   **Report:** ví dụ resolve `{{var.*}}` ở cả 2 scope.
### Bước 1.4 — Fetch Builder: parse curl + template/flow store
*   **Map SPEC:** \[SYS\] 5, 10.1, 10.3, 4.3; \[REQ\] 1.5 (Fetch Builder), 7.1.
*   **Kết quả code cần có:**
    *   `backend/src/modules/templates/parse-curl.ts`
    *   `backend/src/modules/templates/*` (CRUD flow gồm `steps[]`, `inputs[]` với source runtime/store/context)
*   **DoD:** paste curl → sinh template đúng (method/url/headers/body/placeholder); lưu/sửa/xóa flow nhiều step.
*   **Report:** ví dụ curl → template JSON.
### Bước 1.5 — Fetch Executor (sequential flow + extraction)
*   **Map SPEC:** \[SYS\] 10.1, 10.2, 4.2; \[REQ\] 1.2, 7.1.
*   **Kết quả code cần có:**
    *   `backend/src/engine/executor.ts` (chạy steps tuần tự, shared context, stopOnError)
    *   `backend/src/engine/extract.ts` (JSONPath nested/mảng/wildcard, pinToVar)
*   **DoD:**
    *   Flow 2 step (lấy token → gọi API 2 dùng `{{ctx.auth.accessToken}}`) chạy end-to-end.
    *   Extraction JSONPath đa cấp + `[*]` đúng; `pinToVar` ghi vào rtdb-variables.
    *   Step lỗi → dừng + ghi log chi tiết.
*   **Report:** log 1 flow chạy đủ 2 step + giá trị extract + biến được pin.
### Bước 1.6 — History & Logs API
*   **Map SPEC:** \[SYS\] 3.2, 3.3, 4.4; \[REQ\] 1.3, 1.4.
*   **Kết quả code cần có:**
    *   `backend/src/modules/history/*`, `backend/src/modules/logs/*` (filter theo service/business/level/success)
*   **DoD:** mỗi lần execute ghi history + log; filter trả đúng; log không chứa plaintext token.
*   **Report:** ví dụ query filter + xác nhận không lộ secret trong log.
### Bước 1.7 — Issues API + Markdown export
*   **Map SPEC:** \[SYS\] 3.4, 4.5; \[REQ\] 1.6, 1.7.
*   **Kết quả code cần có:**
    *   `backend/src/modules/issues/*` (CRUD, `GET /:id/markdown`)
*   **DoD:** CRUD bug/issue; endpoint markdown trả đúng format mẫu.
*   **Report:** ví dụ issue → markdown output.
### Bước 1.8 — Extracted Data API
*   **Map SPEC:** \[SYS\] 10.5; \[REQ\] 7.2.
*   **Kết quả code cần có:** `backend/src/modules/extractions/*` (GET filter owner/service, kèm template nguồn + thời điểm).
*   **DoD:** trả đúng giá trị đã trích (repo name, git link, cloudflare id...) + nguồn + timestamp.
*   **Report:** ví dụ danh sách extraction.

**DoD chung Backend (Phase 1):** toàn bộ endpoint mục \[SYS\] 4 + 10.4/10.5 hoạt động; response chuẩn `{ ok, data?, error? }`; không endpoint nào trả plaintext credential; unit test core (crypto, placeholder, extract, executor) pass; coverage core ≥ 80%.

## Phase 2 — Frontend
> Mọi bước FE phải tuân thủ \[UI\] mục 0–10 (tokens, modal rules, tooltip, scrollbar, theme, font mảnh). DoD mỗi bước FE gồm cả **tuân thủ UI checklist** tương ứng.
### Bước 2.1 — Design system nền + theme
*   **Map SPEC:** \[UI\] 1, 2, 5, 4.8; \[REQ\] 5.
*   **Kết quả code cần có:** `frontend/src/styles/tokens.css` (CSS variables), theme toggle (`data-theme`), scrollbar mảnh, base layout.
*   **DoD:** đổi theme sáng/tối không hardcode màu; font-weight 300–400, base 13px, spacing 4px; scrollbar mảnh đồng bộ theme.
*   **Report:** ảnh 2 theme + xác nhận token dùng ở mọi nơi.
### Bước 2.2 — Component library (Button, Modal, Tooltip, Form, Table)
*   **Map SPEC:** \[UI\] 4.1–4.7.
*   **Kết quả code cần có:** `frontend/src/components/{Button,Modal,Tooltip,Field,Table}.tsx`.
*   **DoD:**
    *   Modal: có nút X, **click ngoài KHÔNG đóng**, tự scrollbar khi tràn (`max-height:90vh`+overflow), trap focus.
    *   KHÔNG dùng `alert/confirm/prompt` browser ở bất kỳ đâu → NotificationModal + ConfirmModal.
    *   Mọi Button có icon + tooltip mô tả chức năng & kết quả.
*   **Report:** demo modal (click ngoài không đóng), grep xác nhận 0 `alert(`.
### Bước 2.3 — Layout khung (Topbar/Sidebar/Content) + responsive
*   **Map SPEC:** \[UI\] 3, 2; \[REQ\] 5.
*   **Kết quả code cần có:** `frontend/src/layout/*`, owner selector + theme toggle ở Topbar.
*   **DoD:** responsive mobile(drawer)/tablet/desktop cùng 1 hệ component; content scroll riêng, không scroll ngang.
*   **Report:** ảnh 3 breakpoint.
### Bước 2.4 — Màn Credential/Owner + import/export
*   **Map SPEC:** \[UI\] 4.7; \[SYS\] 4.1; \[REQ\] 2.1.
*   **Kết quả code cần có:** `frontend/src/pages/Credentials/*`.
*   **DoD:** list masked + nút 👁 mở confirm trước khi lộ; thêm nhiều giá trị 1 key; import/export qua modal.
*   **Report:** ảnh masked + flow lộ giá trị qua confirm.
### Bước 2.5 — Fetch Flow Builder (multi-step)
*   **Map SPEC:** \[UI\] 10.1, 10.2, 10.6; \[SYS\] 5, 10.1.
*   **Kết quả code cần có:** `frontend/src/pages/FetchBuilder/*` (steps pane + step editor + extract block + placeholder autocomplete/highlight + transform pipe builder + advanced JS editor modal có badge sandbox + nút Test).
*   **DoD:** thêm/xóa/sắp xếp step; placeholder highlight + autocomplete `{{ctx/var/input}}`; JS editor test ra kết quả qua modal (không alert).
*   **Report:** ảnh builder + demo autocomplete + test JS.
### Bước 2.6 — Execute Modal + step progress
*   **Map SPEC:** \[UI\] 10.3; \[SYS\] 10.1, 10.3.
*   **Kết quả code cần có:** `frontend/src/features/execute/*`.
*   **DoD:** input runtime hỏi qua modal (scroll khi nhiều field); confirm khi flow có side-effect; stepper progress pending/running/success/error; step lỗi xem log.
*   **Report:** ảnh stepper + 1 lần chạy lỗi có log.
### Bước 2.7 — View History & Logs
*   **Map SPEC:** \[UI\] 4.7; \[SYS\] 4.4.
*   **DoD:** filter theo service/business/level; xem chi tiết log.
*   **Report:** ảnh filter.
### Bước 2.8 — Inspect Element Mode + quản lý Issue
*   **Map SPEC:** \[UI\] 6; \[SYS\] 3.4, 4.5; \[REQ\] 1.6.
*   **Kết quả code cần có:** `frontend/src/features/inspect/*`.
*   **DoD:** toggle inspect; hover highlight; chọn **nhiều element**; form tạo issue kèm info element; CRUD issue; export/copy Markdown.
*   **Report:** ảnh chọn nhiều element → issue → markdown.
### Bước 2.9 — View Extracted Data + View Variables
*   **Map SPEC:** \[UI\] 10.4, 10.5; \[SYS\] 10.4, 10.5; \[REQ\] 7.2, 7.3.
*   **DoD:** Extracted Data filter + pin thành biến (confirm); Variables tab global/owner, CRUD qua modal, copy `{{var.*}}`.
*   **Report:** ảnh 2 view + flow pin biến.

**DoD chung Frontend (Phase 2):** đủ 11 mục checklist \[UI\] 9 + checklist \[UI\] 10.7; 0 `alert/confirm/prompt`; chạy responsive 3 breakpoint; 2 theme hoạt động.

## Phase 4 — Test & QA
### Bước 4.1 — Unit test
*   **Map SPEC:** \[SYS\] 9, 10.7.
*   **Kết quả cần có:** test cho crypto, placeholder/transform, sandbox, extract JSONPath, executor, parse-curl.
*   **DoD:** coverage core ≥ 80%; sandbox test chứng minh chặn network/fs + timeout.
*   **Report:** coverage report (bảng %).
### Bước 4.2 — Test case (integration/E2E)
*   **Map SPEC:** \[SYS\] 9 (acceptance), \[UI\] 9.
*   **Kết quả cần có:** `docs/TEST_CASES.md` (bảng: id, mô tả, bước, kỳ vọng, map acceptance) + E2E cho luồng chính + flow 2 step.
*   **DoD:** mọi acceptance criteria \[SYS\] 9 & 10.7 có ít nhất 1 test case pass.
*   **Report:** bảng pass/fail từng test case.
### Bước 4.3 — Smoke data + smoke test
*   **Kết quả cần có:** `scripts/seed-smoke.ts` tạo owner mẫu, credential giả (KHÔNG dùng secret thật đã lộ), template flow mẫu; `docs/SMOKE.md`.
*   **DoD:** chạy seed → app hiển thị dữ liệu mẫu → smoke test luồng chính xanh.
*   **Report:** log smoke test + ảnh app có data mẫu.
